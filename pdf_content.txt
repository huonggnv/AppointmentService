Hạ tầng chung — áp dụng cho tất cả 7 đề tài 
 
Tất cả 7 đề tài đều sử dụng chung một kiến trúc microservices chuẩn. Mỗi đề tài được phân chia 
thành 3 nhóm sinh viên, mỗi nhóm xây dựng và vận hành một service độc lập hoàn toàn. Các 
service giao tiếp với nhau thông qua API Gateway 
 
Thành phần Công nghệ Vai trò & Trách nhiệm 
API Gateway ASP .NET core API Điểm vào duy nhất: xác thực JWT, định tuyến 
request tới đúng service, rate limiting 
Frontend VueJS 3 + Vuetify 3 1 project VueJS dùng chung, gọi API qua Gateway, 
quản lý state với Pinia 
Database SQL Server (mỗi service 1 
DB riêng) 
Database per service: service nào cần dữ liệu của 
service khác thì giao tiếp qua API hoặc message 
Container Docker + docker-compose Mỗi service chạy trong container riêng, compose file 
chạy toàn bộ hệ thống 
Auth JWT Bearer Token 1 service duy nhất cấp token, các service còn lại chỉ 
validate token bằng shared secret 
API Docs Swagger / OpenAPI Mỗi service có Swagger UI riêng, tài liệu hóa đầy đủ 
endpoint 
 
Quy tắc thiết kế Microservice cho sinh viên 
• Database per service: mỗi service có database SQL Server riêng, KHÔNG share database 
với service khác 
• Không foreign key chéo service: thay vì FK cứng, lưu Id của bản ghi bên kia dưới dạng 
reference ID (plain column) 
• Giao tiếp đồng bộ (REST): gọi API qua Gateway khi cần dữ liệu real-time, ví dụ Order 
Service gọi Product Service để lấy giá 
• JWT do 1 service duy nhất cấp (User/Auth Service), các service còn lại chỉ validate token 
bằng shared secret 
• Mỗi service deploy bằng Docker container độc lập, chạy được kể cả khi service khác bị 
down

ĐỀ TÀI 01: Hệ thống quản lý bán hàng & kho hàng 
 
1. Mô tả đề tài 
Nền tảng quản lý toàn bộ hoạt động bán lẻ nội bộ cho cửa hàng hoặc doanh nghiệp vừa và nhỏ 
theo mô hình microservices. Ba nhóm độc lập phụ trách: quản lý danh mục sản phẩm và kho, xử 
lý đơn hàng và giao dịch bán, quản lý người dùng và báo cáo tổng hợp. 
 
2. Bài toán & lý do chọn Microservices 
Cửa hàng vừa và nhỏ quản lý kho và bán hàng thủ công bằng Excel dễ dẫn đến sai lệch tồn kho, 
thiếu hàng đột xuất và khó theo dõi doanh thu. Việc tách thành microservices giúp mỗi nhóm phát 
triển và deploy độc lập, tăng khả năng mở rộng khi lượng đơn hàng tăng. 
 
3. Phân quyền người dùng 
• Admin: quản lý toàn hệ thống, xem báo cáo tổng hợp — truy cập qua API Gateway 
• Nhân viên bán hàng: tạo đơn bán, tra cứu sản phẩm — JWT role Sales 
• Thủ kho: quản lý nhập hàng, cập nhật tồn kho — JWT role Warehouse 
 
4. Kiến trúc Microservices tổng quan 
 
API Gateway Ocelot API Gateway — điểm vào duy nhất, định tuyến request tới đúng 
service, xác thực JWT trước khi forward

Inventory Service 
Quản lý sản phẩm, danh mục, tồn kho, cảnh báo hết hàng, nhập kho từ nhà 
cung cấp 
Nhóm 2 — Order & 
Sales Service 
Xử lý đơn bán hàng, quản lý khách hàng, nhà cung cấp, công nợ 
Nhóm 3 — User &

ĐỀ TÀI 02: Hệ thống quản lý thư viện số 
 
1. Mô tả đề tài 
Ứng dụng quản lý thư viện trường học theo kiến trúc microservices với ba service độc lập: 
Catalog Service quản lý đầu sách và số bản sao, Circulation Service xử lý mượn/trả và tính phạt, 
Identity Service quản lý tài khoản độc giả và thống kê. Sinh viên học cách tách nghiệp vụ thư 
viện thành các domain riêng biệt. 
 
2. Bài toán & lý do chọn Microservices 
Thư viện truyền thống gặp khó khăn khi số lượng độc giả lớn: tìm kiếm sách chậm, theo dõi 
mượn/trả thủ công, tính phạt sai. Kiến trúc microservices cho phép mở rộng riêng từng service — 
ví dụ scale Catalog Service khi nhu cầu tìm kiếm tăng cao. 
 
3. Phân quyền người dùng 
• Admin: quản lý danh mục sách, cấu hình quy tắc mượn trả, xem báo cáo tổng hợp 
• Thủ thư: xử lý phiếu mượn/trả, tra cứu tình trạng, xác nhận trả sách 
• Độc giả: tìm kiếm sách, xem lịch sử mượn, trạng thái phiếu mượn hiện tại

ĐỀ TÀI 04: Hệ thống quản lý khóa học & học viên trung tâm 
 
1. Mô tả đề tài 
Nền tảng quản lý trung tâm đào tạo (ngoại ngữ, tin học, kỹ năng) theo kiến trúc microservices. Ba 
service: quản lý khóa học và lịch học, quản lý học viên và điểm danh, quản lý học phí và kết quả. 
Sinh viên học cách tách nghiệp vụ giáo dục thành các bounded context rõ ràng. 
 
2. Bài toán & lý do chọn Microservices 
Trung tâm đào tạo với hàng trăm học viên không thể quản lý thủ công: sắp xếp lớp, điểm danh 
từng buổi, thu học phí và thông báo kết quả mất nhiều công sức. Kiến trúc microservices cho 
phép mỗi team phát triển độc lập từng phần của hệ thống. 
 
3. Phân quyền người dùng 
• Admin: quản lý toàn bộ khóa học, giáo viên, báo cáo tài chính 
• Giáo viên: xem lớp mình phụ trách, điểm danh, nhập kết quả kiểm tra 
• Học viên: xem lịch học, tỷ lệ chuyên cần, kết quả và học phí của mình

ĐỀ TÀI 05: Hệ thống đặt lịch & quản lý phòng khám

2. Bài toán & lý do chọn Microservices 
Phòng khám nhỏ xếp lịch thủ công dễ trùng giờ, hồ sơ bệnh án giấy khó tra cứu. Với 
microservices, Appointment Service có thể scale riêng trong giờ cao điểm đặt lịch, còn Medical 
Record Service chạy ổn định phục vụ bác sĩ tra cứu bệnh án.

• Admin: quản lý bác sĩ, danh mục thuốc, cấu hình hệ thống, báo cáo tài chính 
• Bác sĩ: xem lịch hẹn hôm nay, khám bệnh, ghi bệnh án, kê đơn thuốc 
• Y tá/Tiếp tân: tiếp nhận bệnh nhân, xếp hàng chờ, thu viện phí 
• Bệnh nhân: đặt lịch khám, xem lịch sử khám, đơn thuốc cũ

ĐỀ TÀI 06: Hệ thống quản lý dự án & phân công công việc 
 
1. Mô tả đề tài 
Công cụ quản lý dự án và task nội bộ theo mô hình Kanban/Scrum đơn giản với kiến trúc 
microservices. Ba service: quản lý dự án và thành viên, quản lý task và kanban board, quản lý 
bình luận và thông báo. Sinh viên học cách thiết kế hệ thống cộng tác theo domain. 
 
2. Bài toán & lý do chọn Microservices 
Nhóm làm việc không có công cụ tập trung dẫn đến không biết ai đang làm gì, deadline nào sắp 
đến. Kiến trúc microservices cho phép tách biệt rõ ràng: Project Service quản lý quyền truy cập, 
Task Service xử lý logic Kanban, Notification Service phát thông báo. 
 
3. Phân quyền người dùng 
• Project Manager: tạo và quản lý dự án, thêm thành viên, xem toàn bộ tiến độ 
• Member/Developer: xem task được giao, cập nhật trạng thái, bình luận, log thời gian 
• Viewer: xem tiến độ dự án nhưng không chỉnh sửa task 
 
4. Kiến trúc Microservices tổng quan 
 
API Gateway Ocelot API Gateway — điểm vào duy nhất, định tuyến request tới đúng 
service, xác thực JWT trước khi forward 
Nhóm 1 — Project & 
Member Service 
Tạo dự án, quản lý thành viên với vai trò, sprint và milestone 
Nhóm 2 — Task & 
Kanban Service 
CRUD task, kanban board, sub-task, log thời gian làm việc 
Nhóm 3 — Comment 
& Notify Service 
JWT auth, bình luận vào task, nhật ký hoạt động, thông báo 
 
Hạ tầng bổ sung: Mỗi nhóm: 1 ASP.NET Core 8 Web API · 1 SQL Server database riêng · 
Docker container độc lập · Swagger UI

ĐỀ TÀI 07: Hệ thống quản lý ký túc xá sinh viên 
 
1. Mô tả đề tài 
Ứng dụng quản lý ký túc xá trường đại học theo kiến trúc microservices. Ba service phụ trách: 
quản lý tòa nhà và phòng ở, xử lý hợp đồng và đăng ký phòng, quản lý thu tiền và xử lý sự cố. 
Sinh viên học cách tách hệ thống quản lý tài sản thành các service có trách nhiệm rõ ràng. 
 
2. Bài toán & lý do chọn Microservices 
Ký túc xá trường học với hàng nghìn sinh viên không thể quản lý phòng, hợp đồng và thu tiền 
bằng Excel. Tách thành microservices giúp nhân viên ký túc xá, kế toán và ban quản lý làm việc 
trên các phần hệ thống riêng biệt mà không ảnh hưởng lẫn nhau. 
 
3. Phân quyền người dùng 
• Quản lý KTX (Admin): toàn quyền quản lý tòa nhà, phòng, hợp đồng, báo cáo tài chính 
• Nhân viên: xử lý đăng ký phòng, thu tiền, xử lý yêu cầu sửa chữa 
• Sinh viên: đăng ký phòng, xem hợp đồng, trạng thái thanh toán, gửi sự cố 
 
4. Kiến trúc Microservices tổng quan 
 
API Gateway Ocelot API Gateway — điểm vào duy nhất, định tuyến request tới đúng 
service, xác thực JWT trước khi forward 
Nhóm 1 — Room & 
Building Service 
CRUD tòa nhà, loại phòng, trạng thái phòng, sơ đồ phòng theo tầng 
Nhóm 2 — Contract & 
Student Service 
Hồ sơ sinh viên, đăng ký phòng online, hợp đồng thuê phòng 
Nhóm 3 — Billing & 
Maintenance Service 
JWT auth, thu tiền phòng hàng tháng, xử lý yêu cầu sửa chữa 
 
Hạ tầng bổ sung: Mỗi nhóm: 1 ASP.NET Core 8 Web API · 1 SQL Server database riêng · 
Docker container độc lập · Swagger UI
